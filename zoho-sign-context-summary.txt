9. Business-process recommendation derived from this session

For onsite service-signing flows, the preferred architecture is:

- Feishu spreadsheet or CRM as source of pre-known fields
- Engineer-side confirmation layer for onsite-only facts
- Zoho Sign as the signing execution layer

Recommended data split:

- System-prefilled fields:
  customer identity, address, device model, serials, prior case data

- Engineer-confirmed onsite fields:
  actual findings, work completed, replacement info, final status, notes, confirmed recipient email

- Zoho-generated signing fields:
  signature, signing timestamp, signing status, audit trail

Best practice:

- Do not directly sign whatever is in Feishu at that moment.
- Build a signing snapshot from:
  1. Feishu source data
  2. Engineer confirmation data
  3. Recipient confirmation

Then send that snapshot to Zoho Sign.

12.6 Formatting rules confirmed for the formal mapping layer

Two rules were explicitly decided for the formal script:

- Feishu `日期` is stored as a timestamp.
- When mapped into Zoho `service_date`, it should be converted into German
  local date format:
  DD.MM.YYYY
  example:
  20.01.2026

- Feishu `周数 KW` should not be sent as the full original string when it
  contains a prefix such as `KW07`.
- Only the integer week value should be sent to Zoho.
  examples:
  `KW07` -> `7`
  `KW16` -> `16`

14. Feishu -> own server trigger design note

One later design question was discussed:

- if a Feishu Bitable workflow / automation sends a request to the user's own
  server, can it also reliably include which user created or triggered the
  request

Current practical conclusion:

- this should not be assumed to be reliably provided by default in every
  trigger payload
- the safer design is to send at least:
  - record_id
  - table_id or other local routing metadata if useful
- then let the server call Feishu Bitable API again to read the full record

Recommended implementation pattern:

1. Feishu sends a request to the server with a stable record identifier.
2. The server queries the record through Feishu API.
3. The server reads the user identity from a record field such as:
   - 创建人
   - 提交人
   - 工程师
   - or a custom stored open_id field

So the current recommendation is:

- do not rely only on the trigger payload to carry the operator identity
- keep the creator / submitter identity in the table itself whenever that
  identity matters downstream
- use server-side follow-up queries for correctness

- sign_server.py

Current HTTP behavior:

- GET / or /health
  returns a simple health JSON
- POST /sign/start
  accepts JSON with:
  - wo
  - optional notify_open_id

Current server assumption:

- upstream caller only needs to send the work-order number (`WO`)
- server itself searches the Feishu table by `上门单号`

Current server flow:

1. receive `WO`
2. fetch Feishu record by work-order number
3. map Feishu fields into Zoho fields using the latest confirmed mapping
4. normalize:
   - `日期` -> `DD.MM.YYYY`
   - `周数 KW` -> integer only
5. validate Zoho-required fields
6. validate that mapped values do not contain Chinese characters
7. if validation fails:
   send detailed failure text to a Feishu user by open_id
   (currently defaults to Marco)
8. if validation passes:
   create Zoho embedded signing request
   with empty `recipient_name` and empty `recipient_email`
9. fetch the embedded signing link from Zoho
10. send the link to the Feishu user by bot message

Validation rules now enforced by server:

- all current Zoho required mapped fields must be non-empty
- mapped values must not contain Chinese

Current default Feishu notify target:

- Marco open_id:
  `ou_59c3ace2665789eeea82ffda54b21bc6`

The server was started locally and tested with:

- POST /sign/start
- wo = `WO-106915`

Result:

- server route worked
- Feishu lookup worked

- if Feishu data is not yet safe for Zoho submission,

17.2 Interface update on 2026-04-16

The server trigger contract has now been standardized.

Preferred request body for `POST /sign/start` is now:

- `record_id`
- `trigger_open_id`

Optional fields:

- `notify_open_id`
- `wo`

Interpretation:

- `record_id` is now the primary lookup key
- server fetches the Feishu row directly by `record_id`
- `trigger_open_id` is the default Feishu notification target
- `notify_open_id` can override that target when needed
- `wo` is now only a compatibility/debug field and is not used as the main lookup key

Updated practical flow:

1. Feishu automation sends `record_id` and `trigger_open_id`
2. server reads the record by `record_id`
3. server maps and validates fields
4. if invalid, server sends detailed Feishu error message to the notification target
5. if valid, server creates the Zoho embedded sign request
6. server gets the embedded sign link
7. server sends the sign link back by Feishu message


17.3 Trigger authentication requirement

The server now requires a shared trigger authentication token for
`POST /sign/start`.

Accepted request headers:

- `X-Trigger-Token: <token>`
- or `Authorization: Bearer <token>`

Deployment rule:

- the real token value must be configured through Render environment variable
  `TRIGGER_AUTH_TOKEN`
- the real secret should not be committed into the repository

19. Current integrated server status on 2026-04-16

The project later moved from separate temporary scripts toward one deployed
Render service with two HTTP routes:

- `POST /sign/start`
- `POST /webhooks/zoho-sign`


19.1 Current `/sign/start` behavior

The Feishu-trigger route now:

- accepts:
  - `record_id`
  - `trigger_open_id`
- reads the Feishu record by `record_id`
- maps the current Feishu fields into the Zoho template
- validates:
  - required fields are present
  - mapped values do not contain Chinese characters
- creates a Zoho embedded signing request
- requests the Zoho embed token / link
- sends the signing link back by Feishu bot message

Security for this route:

- protected by shared secret
  `TRIGGER_AUTH_TOKEN`


19.2 Current `/webhooks/zoho-sign` behavior

The Zoho webhook route now:

- accepts Zoho callback requests
- verifies the Zoho HMAC signature
- extracts `request_id`
- resolves the matching Feishu record
- downloads the signed PDF from Zoho
- uploads the PDF into Feishu
- writes the uploaded file into the Feishu attachment field

Security for this route:

- protected by Zoho webhook HMAC secret
  `ZOHO_WEBHOOK_SECRET`


19.3 Important webhook implementation correction

An earlier webhook verification attempt returned:

- HTTP 401

because the signature verification format was initially implemented
incorrectly.

Correct interpretation:

- Zoho webhook signature must be verified as:
  `base64(HMAC-SHA256(raw_body))`

This was later corrected in the deployed server implementation.


19.4 Current request mapping strategy

The current server stores request mapping locally in:

- `request_map.json`

The stored relationship is:

- `zoho request_id`
- `feishu record_id`
- `wo`
- `action_id`

This is currently enough for webhook callback resolution, but it is still
considered a replaceable temporary persistence layer.

The user may later replace this with another mapping approach.
